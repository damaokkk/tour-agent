# SmartTour 项目部署总结

## 项目概述

**SmartTour** 是一个智能旅游规划助手，基于 AI 技术为用户提供个性化的旅游行程规划服务。

### 技术栈
- **前端**: React + TypeScript + Vite + Tailwind CSS
- **后端**: Node.js + Express
- **AI 服务**: 火山方舟（豆包大模型）
- **搜索服务**: Tavily API
- **部署平台**: Vercel（前端）+ Railway（后端）

---

## 部署架构

```
用户 → Vercel（前端）→ Railway（后端）→ 火山方舟 API
                          ↓
                      Tavily API
```

### 生产环境地址
- **前端**: https://tour-agent-seven.vercel.app
- **后端**: https://tour-agent-production.up.railway.app

## 部署流程

### 步骤 1：准备 Git 仓库

**操作**:
1. 创建 GitHub 仓库 `tour-agent`
2. 初始化本地 Git 仓库
3. 创建 `.gitignore` 文件排除 `node_modules` 和敏感文件
4. 推送代码到 GitHub

**遇到的问题**:
- SSH 密钥权限问题（Deploy Key 无法推送）
- 解决方案：改用 HTTPS 方式，或使用 SSH agent 指定正确的密钥

```powershell
# 使用 SSH 密钥推送
$env:GIT_SSH_COMMAND="ssh -i C:/Users/15307/.ssh/id_ed25519"
git push origin main
```

---

### 步骤 2：部署前端到 Vercel

**操作**:
1. 注册 Vercel 账号（使用 GitHub 登录）
2. 导入 GitHub 仓库
3. 配置项目：
   - Framework Preset: `Vite`
   - Root Directory: `frontend`
4. 部署

**遇到的问题与解决**:

#### 问题 1：找不到 `frontend` 目录
**错误信息**:
```
sh: line 1: cd: frontend: No such file or directory
```

**原因**: Vercel 默认在项目根目录运行命令，但前端代码在 `frontend` 子目录。

**解决方案**: 在 Vercel 项目设置中配置 **Root Directory** 为 `frontend`。

#### 问题 2：`tsc` 命令权限拒绝
**错误信息**:
```
sh: line 1: /vercel/path0/frontend/node_modules/.bin/tsc: Permission denied
```

**原因**: Vercel 的 Linux 环境对 `.bin` 目录下的命令文件权限有问题。

**解决方案**: 
1. 使用 `npx tsc` 代替 `tsc`
2. 最终方案：跳过 `tsc`，直接使用 `vite build`（Vite 本身支持 TypeScript）

**package.json 修改**:
```json
{
  "scripts": {
    "build": "node ./node_modules/vite/bin/vite.js build"
  }
}
```

#### 问题 3：`vite` 命令权限拒绝
**错误信息**:
```
sh: line 1: /vercel/path0/frontend/node_modules/.bin/vite: Permission denied
```

**解决方案**: 使用 Node 直接运行 vite 的入口文件：
```json
{
  "scripts": {
    "build": "node ./node_modules/vite/bin/vite.js build"
  }
}
```

---

### 步骤 3：部署后端到 Railway

**操作**:
1. 注册 Railway 账号
2. 创建新项目，选择 GitHub 仓库
3. 配置环境变量（API Keys）
4. 配置启动命令
5. 部署

**遇到的问题与解决**:

#### 问题 1：Railway 无法识别项目类型
**错误信息**:
```
⚠ Script start.sh not found
✖ Railpack could not determine how to build the app.
```

**原因**: 项目有 `backend` 和 `frontend` 两个目录，Railway 不知道从哪里开始构建。

**解决方案**: 创建 `railway.json` 配置文件，明确指定构建计划：

```json
{
  "$schema": "https://railway.app/railway.schema.json",
  "build": {
    "builder": "NIXPACKS",
    "nixpacksPlan": {
      "providers": ["node"],
      "phases": {
        "setup": {
          "nixPkgs": ["nodejs"]
        },
        "install": {
          "cmds": ["cd backend && npm install"]
        },
        "build": {
          "cmds": []
        }
      }
    }
  },
  "deploy": {
    "startCommand": "cd backend && node src/index.js",
    "healthcheckPath": "/",
    "healthcheckTimeout": 100
  }
}
```

#### 问题 2：CORS 跨域问题
**原因**: 前端域名和后端域名不同，浏览器阻止跨域请求。

**解决方案**: 在 `backend/src/index.js` 中添加前端域名到 CORS 白名单：

```javascript
app.use(cors({
  origin: [
    'http://localhost:5173',
    'https://tour-agent-seven.vercel.app'  // 添加生产环境域名
  ],
  credentials: true
}));
```

## 核心问题总结

### 1. 跨平台构建问题
**现象**: Windows 本地开发正常，Linux 部署环境报错

**根本原因**: 
- Windows 和 Linux 的文件权限机制不同
- Windows 使用 `\` 作为路径分隔符，Linux 使用 `/`
- `node_modules/.bin` 下的命令在 Linux 上需要执行权限

**解决思路**:
- 避免直接使用 `.bin` 目录下的命令
- 使用 `node` 直接运行模块入口文件
- 使用 `npx` 或完整路径

### 2. 多目录项目结构问题
**现象**: 部署平台无法自动识别项目结构

**根本原因**: 项目采用前后端分离的 monorepo 结构，但部署平台期望单一项目结构

**解决思路**:
- 使用配置文件（`vercel.json`, `railway.json`）明确指定构建参数
- 配置 Root Directory
- 指定启动命令和工作目录

### 3. 环境变量和配置管理
**关键配置**:
- API Keys（OpenAI/Tavily）通过环境变量注入
- 前端通过 `import.meta.env.PROD` 区分开发和生产环境
- CORS 配置需要同时包含开发和生产域名

## 经验总结

### 关键学习点
1. **跨平台兼容性**: 开发环境和生产环境的差异需要特别注意
2. **配置文件优先**: 使用明确的配置文件比依赖自动检测更可靠
3. **逐步验证**: 每个步骤完成后立即验证，避免问题累积
4. **日志分析**: 仔细阅读错误日志，定位问题根源

### 最佳实践
1. 使用 `npx` 或 `node` 直接运行命令，避免权限问题
2. 为每个部署平台创建专门的配置文件
3. 环境变量通过平台配置，不提交到代码仓库
4. CORS 配置要包含所有可能的访问域名
